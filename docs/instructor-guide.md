# 講師・先輩向け運用メモ

この資料は、実習を進行する講師・先輩向けです。

参加者向けの手順は [workshop.md](workshop.md) にまとめています。

## 実習前に確認すること

- 参加者がGitHubリポジトリにアクセスできる
- 必要な権限が付与されている
- branch protection ruleを設定するか決めている
- `main` への直接pushを誰ができるか把握している
- 参加者がVS CodeとGitを使える状態になっている

## 最初に説明すること

- VS Code左下のブランチ名を毎回確認する
- 個人作業では `main` にpushしない
- `Sync Changes` はpullとpushの両方を含むことがある
- 秘密情報や個人情報をcommitしない
- 困ったら作業を止めて画面を見せる

## PR練習の進め方

前半では、参加者ごとに `introductions/github-id.md` を追加してもらいます。

各自が別ファイルを編集するため、基本的にコンフリクトは起きません。

確認すること:

- branch名が `github-id_dev` になっている
- PRのbaseが `main`
- PRのcompareが参加者の作業ブランチ
- Source Controlに未コミットの変更が残っていない

## コンフリクト練習の進め方

後半では、複数人に `practice/conflict-message.txt` の同じ行を編集してもらいます。

進行例:

1. 全員が `github-id_conflict` ブランチを作る
2. 同じ1行目を編集する
3. 全員がPRを作る
4. 先に1人のPRだけmergeする
5. 残りの参加者に `main` を取り込ませる
6. コンフリクト解消を確認する

## mainにpushしてしまった場合

参加者に追加操作をさせず、まず状況を確認します。

確認すること:

- どのcommitがpushされたか
- 秘密情報や個人情報が含まれていないか
- 他の参加者がすでにpullしていないか
- revertで戻すのか、別PRで修正するのか

`git push --force` は原則として参加者に実行させません。

## 秘密情報をpushしてしまった場合

履歴削除より先に、漏れたキーやトークンを無効化します。

初動:

1. 作業を止める
2. 何がpushされたか確認する
3. APIキーやトークンを無効化または再発行する
4. 必要に応じて履歴からの削除を検討する
