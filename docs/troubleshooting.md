# 困ったとき

迷ったら、追加の操作をせずに講師または先輩に画面を見せてください。

特に、作業を消す操作や履歴を書き換える操作は自分だけで進めないでください。

## push前に確認すること

- VS Code左下のブランチ名が自分の作業ブランチになっている
- `main` ではない
- Source Control に未コミットの変更が残っていない
- GitHubにpushしようとしているブランチ名が自分の作業ブランチ

## mainで作業していることに気づいた

まだcommitしていない場合は、変更を消さずに作業ブランチを作ります。

VS Codeの場合:

1. 左下のブランチ名をクリックする
2. `Create new branch` を選ぶ
3. `自分の名前_dev` のようなブランチ名を入力する
4. 左下のブランチ名が新しいブランチになったことを確認する

ターミナルの場合:

```bash
git checkout -b 自分の名前_dev
```

未コミットの変更は、基本的に新しいブランチへ一緒に移動します。

## mainでcommitしてしまったが、まだpushしていない

まだGitHub上の `main` には影響していません。

慌ててpushしないでください。

やること:

1. 作業を止める
2. 講師または先輩に「mainでcommitしたが、まだpushしていない」と伝える
3. VS Code左下のブランチ名とSource Controlの状態を見せる

`git reset --hard` は変更を消す可能性があるため、自分だけで実行しません。

## mainにpushしてしまった

自分だけで直そうとしないでください。

やること:

1. 作業を止める
2. 講師または先輩に「mainにpushしてしまった」と伝える
3. いつpushしたか、何を変更したかを伝える
4. GitHubのcommit画面またはVS Codeの履歴を一緒に確認する

次の操作は、講師の指示なしに実行しません。

- `git reset --hard`
- `git push --force`
- `git revert`

## git pushでログインを求められた

GitHubにログインできていない、または認証設定が必要な状態です。

表示された画面を講師または先輩に見せてください。

## git commitで名前とメールアドレスを求められた

Gitのユーザー情報が未設定です。

表示されたメッセージを講師または先輩に見せて、設定してからもう一度commitします。

## Discard Changesを押しそうになった

`Discard Changes` は、ファイルの変更を消す操作です。

消えてよい変更か確認できるまで押さないでください。

## git push --forceを求められた

`git push --force` はGitHub上の履歴を書き換える操作です。

チーム開発では他の人の作業に影響する可能性があります。

自分だけで実行せず、講師または先輩に相談してください。

## コンフリクト記号が出た

ファイル内に `<<<<<<<`、`=======`、`>>>>>>>` の記号が出たら、コンフリクトしています。

VS Codeの `Accept Current Change`、`Accept Incoming Change`、`Accept Both Changes`、`Compare Changes` を使って確認します。

修正後、これらの記号が残っていないことを確認してください。

## Pull Requestが作れない

次を確認してください。

- 自分の作業ブランチをGitHubにpushしている
- GitHub上で compare が自分のブランチになっている
- base が `main` になっている
- リポジトリへの権限がある

わからない場合は、GitHubの画面を講師または先輩に見せてください。

## 自分のブランチが見つからない

VS Code左下のブランチ名をクリックして、ブランチ一覧を確認します。

GitHubにpush済みなのに見つからない場合は、講師または先輩に画面を見せてください。
