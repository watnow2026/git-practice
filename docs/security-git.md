# セキュリティ観点のGit注意点

この資料では、Git操作でセキュリティ上気をつけることを説明します。

Gitの基本操作やレポジトリ構成とは分けて、「何がなぜ危険なのか」を確認するための資料です。

## まず覚えること

Gitでは、一度GitHubにpushした内容は他の人から見える状態になります。

privateリポジトリでも、チームメンバーや権限を持つ人からは見えます。秘密情報や不要なファイルをcommitしないことが大切です。

## 秘密情報をcommitしない

commitしてはいけないもの:

- パスワード
- APIキー
- アクセストークン
- 秘密鍵
- `.env`
- 本番環境の設定
- 個人情報を含むファイル

なぜダメなのか:

- GitHub上に残る
- 他の人がpullできる
- 削除したcommitの履歴にも残ることがある
- 外部サービスの不正利用につながる

対策:

- `.env` は `.gitignore` に入れる
- サンプルは `.env.example` として置く
- 本物の値はローカルや秘密情報管理サービスで管理する

## git add . は必ず確認してから使う

`git add .` は、現在のディレクトリ以下の変更をまとめてステージングします。

危険な理由:

- 関係ないファイルが入る
- `.env` などの秘密情報を含むファイルが入る
- エディタの個人設定が入る
- 大きなファイルが入る

安全な進め方:

```bash
git status
git add 対象ファイル
git status
```

VS Codeでは、Source Controlで変更ファイル一覧を見て、必要なファイルだけ `+` を押します。

## mainへの直接pushは個人判断でしない

`main` はチーム全員が基準にするブランチです。

危険な理由:

- レビュー前の変更が全員に共有される
- 間違った変更を他の人が取り込む
- 不具合や秘密情報が混ざっても気づきにくい
- どの変更が原因か追いにくくなる

実習中の個人作業では、自分の作業ブランチにpushしてPull Requestを作ります。

## git push --force を単独で実行しない

`git push --force` は、GitHub上の履歴を手元の履歴で上書きする操作です。

危険な理由:

- 他の人のcommitが消えたように見える
- Pull Requestの履歴が壊れる
- 誰が何を変更したか追いにくくなる
- 復旧が難しくなることがある

必要な場面もありますが、初心者は単独で実行しません。講師、リーダー、メンターと一緒に確認してから行います。

## git reset --hard を単独で実行しない

`git reset --hard` は、手元の作業内容を指定したcommitの状態に戻す操作です。

危険な理由:

- 未保存の変更が消える
- 未commitの作業が消える
- 状況によっては戻し方が難しい

実行前に、残す必要がある変更がないか必ず確認します。

## VS Code の Discard Changes に注意する

`Discard Changes` は、ファイルの変更を捨てる操作です。

危険な理由:

- ボタン1つで作業中の変更が消える
- commitしていない内容は戻せないことがある

消えてよい変更だと確認できるまで押さないでください。

## git clean -fd を単独で実行しない

`git clean -fd` は、Gitがまだ管理していない新規ファイルを削除する操作です。

危険な理由:

- 新しく作ったファイルが消える
- まだcommitしていない自己紹介ファイルも対象になることがある
- 一度消すと戻せないことがある

実習中は基本的に使いません。必要に見えても、講師または先輩に相談してください。

## push後の commit --amend に注意する

`git commit --amend` は、直前のcommitを作り直す操作です。

push前なら、commitメッセージの修正などに使うことがあります。

push後に危険な理由:

- GitHub上の履歴と手元の履歴がずれる
- 追加でforce pushが必要になることがある
- 他の人が参照している履歴と合わなくなる

push後にcommitを直したい場合は、新しいcommitを追加するか、講師に相談します。

## 共有済みブランチの rebase に注意する

`rebase` は履歴を整理するために使うことがあります。

共有済みブランチで危険な理由:

- 履歴の形が変わる
- 他の人の手元の履歴とずれる
- force pushが必要になることがある

すでにGitHubにpushしたブランチでrebaseが必要になったら、自分だけで進めずに相談してください。

## リモートブランチ削除に注意する

GitHub上のブランチ削除は、Pull Requestをmergeした後なら行うことがあります。

危険な理由:

- 作業中のブランチを消すと、他の人が確認できなくなる
- 未mergeの作業を見失う可能性がある

PR merge後にGitHubが表示する `Delete branch` は、講師やチームのルールに従って押します。

## 大きなファイルをcommitしない

commitしない方がよいもの:

- 動画
- 大きな画像
- ZIPファイル
- ビルド結果
- 依存パッケージの実体

危険な理由:

- cloneやpullが遅くなる
- レポジトリ全体が重くなる
- 履歴に残り続ける

必要な場合は、Git LFSや外部ストレージを使うことがあります。

## 個人情報を含むデータをcommitしない

commitしてはいけない例:

- 本名、住所、電話番号を含むCSV
- メールアドレス一覧
- ユーザーの本番データ
- スクリーンショットに写った個人情報

なぜダメなのか:

- 権限を持つ人に共有される
- 削除しても履歴に残ることがある
- 法的・運用上の問題になる可能性がある

## もし秘密情報をpushしてしまったら

自分だけで直そうとしないでください。

やること:

1. すぐに作業を止める
2. 講師またはリーダーに伝える
3. 何をpushしたかを伝える
4. その秘密情報を無効化または再発行する
5. 必要に応じて履歴からの削除を検討する

重要なのは、Gitの履歴から消すことより先に、漏れたキーやトークンを無効化することです。

## push前チェックリスト

pushする前に確認します。

- 今いるブランチは自分の作業ブランチか
- `main` にpushしようとしていないか
- `git status` に知らないファイルがないか
- `.env` や秘密情報が含まれていないか
- 大きすぎるファイルが含まれていないか
- Source Controlで変更内容を見たか
- `git add .` で不要なファイルを含めていないか

迷ったら、pushせずに画面を見せて相談してください。
